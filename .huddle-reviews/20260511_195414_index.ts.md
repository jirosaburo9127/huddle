## レビュー (mikan-respond/index.ts)

### バグ/エッジケース漏れ
- **`?? ""` フォールバックが危険**: env 未設定時に `MIKAN_WEBHOOK_SECRET = ""` となる。後段の検証で `header === MIKAN_WEBHOOK_SECRET` のような単純比較を行うと、攻撃者が空ヘッダ or `X-Mikan-Secret: ""` で**認証バイパス可能**。
- 他の必須 env (`ANTHROPIC_API_KEY` 等) は `!` で fail-fast しているのに、これだけ silent fallback で**一貫性が崩れている**。
- diff には宣言だけで**実際の検証ロジックが見えない**。本当に request handler 側で照合しているかフルファイルで要確認 (declared but unused なら #5819 の二の舞)。

### 設計/可読性
- コメントは良い (migration 118 と Vault への参照あり) 。ただし「どこで検証するか」も一言添えるとレビュワーが追いやすい。
- 命名は `MIKAN_WEBHOOK_SECRET` で OK。ヘッダ名 `X-Mikan-Secret` も定数化すると typo 防止になる。

### セキュリティ/入力検証
- **タイミング攻撃**: 単純な `===` ではなく `crypto.timingSafeEqual` 相当 (Deno なら `crypto.subtle` で HMAC 比較、または定数時間比較) を使うべき。
- **空文字許可の禁止**: 起動時に `if (!MIKAN_WEBHOOK_SECRET) throw new Error(...)` で落とす方が安全。
- Vault と同期させる前提なら、**ローテーション手順**もコメント or runbook に残したい (M11 シュナイアー観点)。
- ログに secret やヘッダ値を吐かないことを呼び出し側で要確認。

### 優先度
- **今すぐ**: `?? ""` を `!` または明示的な起動時チェックに変更 / 検証側で timing-safe 比較を採用
- **後で**: ヘッダ名の定数化、ローテーション手順の文書化
- **任意**: コメントに「検証箇所: line XX」を追記

## Codex #2

同意: `?? ""` は指摘どおり危険です。必須 secret なら `!` よりも `const v = ...; if (!v) throw` の明示チェックが一番レビューしやすいです。空文字 secret を許さない意図も残ります。

見落とし候補: 後段の検証が `if (expected && header !== expected)` のような形だと、env 未設定時に検証自体がスキップされる可能性もあります。Claude の「空ヘッダでバイパス」だけでなく「未設定で無認証化」の観点も見るべきです。

過剰気味: タイミング攻撃対策は理想論としては正しいですが、Supabase Edge Function の内部 webhook secret 照合なら、まずは「必須 env」「空文字禁止」「全リクエストで拒否」が優先です。`===` を即ブロッカー扱いするより、低〜中優先の hardening でよいと思います。
