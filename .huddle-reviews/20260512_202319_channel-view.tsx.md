## Codex レビュー

- [優先度: 今すぐ] `handleReturnToJumpTarget` で `executeJump()` の cleanup を `jumpReturnCleanupRef` に保持していますが、通常のジャンプイベントや URL ジャンプ開始時にはこの cleanup が呼ばれていません。戻るジャンプのリトライ/監視が残ったまま別ジャンプすると、後から古い target へスクロール・ハイライトする競合が起き得ます。`executeJump` を呼ぶ全入口の直前、または `executeJump` 冒頭で既存 `jumpReturnCleanupRef.current` を cancel してください。

- [優先度: 今すぐ] `handleReturnToJumpTarget` が `executeJump(jumpReturnTargetId)` を使うことで、通常ジャンプ用の副作用まで再実行されます。`executeJump` が `jumpReturnTargetId` 更新、URL cleanup、イベント用状態更新などを含む場合、戻るボタン押下で戻る先が再設定されたり表示状態が壊れる可能性があります。戻る用途では `executeJump(target, { ... })` に `suppressReturnTarget` / `source: "return"` のような option を追加し、通常ジャンプ専用の副作用を抑止してください。

- [優先度: 後で] cleanup の責務が `jumpReturnCleanupRef` と `stabilizeCleanupRef` に分かれており、どちらが DOM 待ち・どちらが位置安定化を止めるのか読み取りづらくなっています。`returnJumpCleanupRef` など用途を明確にするか、`executeJump` 内で pending jump cleanup を一元管理すると安全です。
