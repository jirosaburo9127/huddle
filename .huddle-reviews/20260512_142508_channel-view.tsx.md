## Codex レビュー

- [優先度: 今すぐ] `INSERT` 購読が `reactions` 全件対象になっており、現在表示中チャンネル以外のリアクションでも毎回 `setMessages(prev.map(...))` が走ります。`newReaction.message_id` が現在の `messages` に存在しない場合は `prev` をそのまま返す、または可能なら購読側で対象メッセージに絞ってください。

- [優先度: 今すぐ] `DELETE` 側は全メッセージを必ず新しいオブジェクトに作り直しています。対象リアクションが無い場合でも全行が再レンダーされるので、`some(r.id === old.id)` があるメッセージだけ更新し、見つからなければ `prev` を返す形にしてください。

- [優先度: 今すぐ] `payload.new as Reaction` は DB 行そのものなので、既存 UI が `Reaction` に `user` / `profiles` など JOIN 済みフィールドを期待している場合に壊れます。`Reaction` 型を DB row 用と表示用で分けるか、リアルタイム追加時に表示で必要な最小フィールドだけに正規化してください。

- [優先度: 後で] 自分の楽観更新との重複排除が `id` のみです。楽観更新時に仮 ID を使っている実装なら、確定 INSERT が別 ID で追加されて二重表示になります。`message_id + user_id + emoji` など実際の一意条件でも重複判定してください。

- [優先度: 後で] 追加/削除イベントの単体テストが欲しいです。特に「対象外メッセージでは state 参照が変わらない」「楽観更新済みリアクションが二重追加されない」「DELETE で対象リアクションだけ消える」を reducer 相当の関数に切り出して検証すると安全です。
