## Codex レビュー

- [優先度: 今すぐ] `releaseTimer` が `stabilizeCleanupRef.current` を無条件に cleanup しているため、古い timer が何らかの理由で残ると新しいジャンプの再固定を解除し得ます。`const cleanup = stabilizeJumpPosition(targetId); stabilizeCleanupRef.current = cleanup;` と保持し、timer 内では `if (stabilizeCleanupRef.current === cleanup) { cleanup(); stabilizeCleanupRef.current = null; }` のように所有者チェックを入れてください。

- [優先度: 後で] `pointerdown` と `mousedown` の両方を登録すると、Pointer Events 対応ブラウザではマウス操作で cleanup が二重に走る可能性があります。cleanup が冪等なら壊れにくいですが、読みやすさと余計な remove を避けるため、`window.PointerEvent` がある場合は `pointerdown` のみ、ない場合だけ `mousedown` を登録する分岐にすると明確です。

- [優先度: 後で] `stabilizeJumpPosition` 側でユーザー操作により自動 cleanup された場合でも `stabilizeCleanupRef.current` が stale な関数のまま残ります。cleanup 内、または呼び出し元のラッパーで `stabilizeCleanupRef.current === cleanup` のとき `null` に戻すと、状態と実態が一致します。

- [優先度: 任意] `JUMP_STABILIZATION_MS` 導入に合わせて、`[120, 260, ... 4200]` のスケジュール配列も定数化すると、5秒前提の値であることが読み取りやすくなります。
