## 呼ばずに確認質問を返すべきケース (重要)
以下のいずれかに当てはまる場合は、\`propose_event\` を呼ばずに、足りない情報を 1〜2 行で
やさしく聞き返してください (例: 「場所はどちらにしましょう？」「日付は何月何日でしょう？」)。

1. **日時が決まっていない**: 「来週どこかで」「今度」「いつかやろう」など具体時刻が無い
2. **日時の解釈が複数ある**: 「金曜」が今週か来週か文脈で確定できない、「3時」が朝か昼か不明 など
3. **過去の日時**: 推定した start_at が現在日時より前になる
   - ただし「昨日の打合せ良かった」のような感想は、そもそも登録不要なので何も呼ばないし聞き返しもしない
4. **場所が話題に上ったのに未確定**: 会話の流れで「どこでやる？」「店どこ？」のように場所が論点に
   なっているのに決まっていない場合 (場所が一切話題に出ていないなら、location 無しで登録して OK)
5. **タイトルが曖昧で文脈にも見当たらない**: 何の予定か推測できない

// 日時を日本語表記に整形 ("5月4日(土) 15:00")
function formatDateTimeJa(iso: string): string {
  const d = new Date(iso);
  if (isNaN(d.getTime())) return iso;
  const jst = new Date(d.getTime() + 9 * 60 * 60 * 1000);
  const month = jst.getUTCMonth() + 1;
  const day = jst.getUTCDate();
  const dow = ["日", "月", "火", "水", "木", "金", "土"][jst.getUTCDay()];
  const h = String(jst.getUTCHours()).padStart(2, "0");
  const m = String(jst.getUTCMinutes()).padStart(2, "0");
  return `${month}月${day}日(${dow}) ${h}:${m}`;
}

function nowJstString(): string {
  return formatDateTimeJa(new Date().toISOString());
}

    // tool_use 優先で処理
    const toolBlock = aiJson.content.find(
      (c): c is Extract<ContentBlock, { type: "tool_use" }> =>
        c.type === "tool_use" && c.name === "propose_event"
    );

    if (toolBlock) {
      const input = toolBlock.input as { title?: unknown; start_at_iso?: unknown; location?: unknown };
      const title = typeof input.title === "string" ? input.title.trim() : "";
      const startIso = typeof input.start_at_iso === "string" ? input.start_at_iso : "";
      const location = typeof input.location === "string" ? input.location.trim() : "";
      const startAt = new Date(startIso);

      if (!title || isNaN(startAt.getTime())) {
        console.warn("[mikan] propose_event invalid input:", input);
      } else {
        const jaDate = formatDateTimeJa(startAt.toISOString());
        const locLine = location ? `\n📍 ${location}` : "";
        const proposalContent =
          `📅 予定の登録を提案します\n\n「${title}」\n${jaDate}${locLine}\n\nこのメッセージにリアクションすると登録します ✅`;

        const { data: msgData, error: insertErr } = await supabase
          .from("messages")
          .insert({
            channel_id: msg.channel_id,
            user_id: MIKAN_USER_ID,
            content: proposalContent,
          })
          .select("id")
          .maybeSingle();

        if (insertErr || !msgData) {
          console.error("[mikan] proposal message insert failed:", insertErr);
          return new Response("insert failed", { status: 500 });
        }

        const { error: propErr } = await supabase.from("event_proposals").insert({
          workspace_id: ch.workspace_id,
          channel_id: msg.channel_id,
          message_id: msgData.id,
          proposed_by: MIKAN_USER_ID,
          for_user_id: msg.user_id,
          title,
          starts_at: startAt.toISOString(),
          location: location || null,
        });

        if (propErr) {
          console.error("[mikan] event_proposals insert failed:", propErr);
        }

        return new Response("proposal posted", { status: 200 });
      }
    }

    // 通常テキスト返信
    const replyText = aiJson.content
      .filter((c): c is Extract<ContentBlock, { type: "text" }> => c.type === "text")
      .map((c) => c.text)
      .join("\n")
      .trim();

    if (!replyText) {
      return new Response("empty reply", { status: 200 });
    }

    const { error: insertErr } = await supabase.from("messages").insert({
      channel_id: msg.channel_id,
      user_id: MIKAN_USER_ID,
      content: replyText,
    });

    if (insertErr) {
      console.error("[mikan] insert reply failed:", insertErr);
      return new Response("insert failed", { status: 500 });
    }

    return new Response("ok", { status: 200 });
  } catch (e) {
    console.error("[mikan] unexpected error:", e);
    return new Response("error", { status: 500 });
  }
});

### 🔒 セキュリティ/入力検証

- **【今すぐ】`title_hint` をそのまま `ilike` に渡している**: `%` や `_` がエスケープされておらず、LLM が悪意ある (or 不規則な) hint を返すと意図しないマッチが発生する。実害は限定的だが、`escape` してから渡すのが原則。RLS は service_role でバイパスしているので、誤って他チャンネルのイベントを引くことは無いが (channel_id でフィルタ済)、念のため `_` `%` `\\` をエスケープ。

- **【今すぐ】LLM の出力 (`title`, `proposalContent`) をそのまま `messages.content` に格納**: チャットの表示側で XSS 対策しているなら問題ないが、フロント側でマークダウン/HTML を rendering している場合は要チェック。最低限 1 行のサニタイズか、表示側でのエスケープ徹底を確認。
- **【後で】`SUPABASE_SERVICE_ROLE_KEY` 利用**: RLS 完全バイパスなので、`channel_id` を信頼している点が前提。webhook 認証が無いと前項と合わせて任意チャンネル書き込みになり、攻撃面として最も大きい。
- **【後で】`web_fetch` の SSRF リスク**: LLM が貼られた URL を fetch する仕様だが、内部 IP (10.x, 169.254.169.254 等) を渡されると Anthropic 側で fetch されるのでホスト側 SSRF にはならない。一方ユーザーが他人の private URL を貼って情報窃取に使う可能性はあるが、これは web_fetch の仕様上避けられない。プロンプトで「公開 URL のみ」を促す or `allowed_domains` 検討。
- **【後で】Web 検索の `blocked_domains` リスト**: Twitter/X だけでなく Note 系まとめサイト等抜けがある。プロンプトで一次情報優先を指示済みなので二重防御として現状で許容、ただし運用で追加更新の余地。
- **【任意】エラー文のクライアント露出**: `ai failed: ${aiRes.status} ${errText}` を 200 で返している (本文に内部エラー)。これは Supabase Webhook 側に返るだけだが、`errText` が API キー等を含むことは無いはずでも、ログのみに留め本文は固定文言が安全。

## 同意/反論

### Codex の新規論点に同意

- **`event_proposals.insert` 失敗時に提案メッセージだけ残る**: これは実害大で同意。「リアクションすると登録します」と表示されているのに、リアクション側のハンドラ (別 Edge Function or trigger) が proposal を引けず無反応になる。**順序を逆転 (proposals 先に insert → message insert) するか、メッセージ削除のロールバックを入れるべき**。Webhook 認証の次に優先で良い。
- **AI API 失敗時の 200 返却**: 同意。Webhook 再試行抑止の意図なら明示コメント、本当に再試行させたいエラー (5xx / rate limit) だけ非200 に分けるのが筋。
- **環境変数の non-null assertion**: 同意。Edge Function 起動時に明示 throw する方が運用ログで原因即判別できる。
- **タイムゾーン情報の喪失**: 同意。`starts_at` が `timestamptz` なら UTC 保存で問題ないが、`event_proposals.starts_at` のカラム型を確認した方が良い。

### 自分の過剰指摘を撤回

- **tool_use の 2 ターン目処理**: Codex の整理に同意。これはコマンド消費型の設計として成立しており、「方針コメント明記で十分」が正しい。レビューでは「コメントで意図を残す」だけに格下げ。
- **LLM 出力の XSS / web_fetch SSRF**: Codex の整理通り、Edge Function のコードレビュー範囲では「確認事項」止まりが妥当。重大扱いは引っ込めます。
- **reminder と propose_event 同時返却**: 低頻度なのでコメント明記で十分に同意。

### 追加の新規論点

- **`mention_type` の検証が無い**: `mentioned_user_id === MIKAN_USER_ID` しか見ていないので、`mention_type` が `channel` 系 broadcast でも mikan が個別反応する可能性。仕様で `user` のみ通すフィルタを入れた方が安全 (将来 `@channel` 実装で大量誤反応を防ぐ)。
- **conversation 配列の `role` 連続問題**: Anthropic API は strict には連続 role を許容するが、`tools` 併用時に user/assistant の最後の role 制約が API バージョンで揺れる。最後が assistant (mikan の自発言) で終わるケースは ctx に msg 自身が含まれるので通常は user 終端だが、保険で「末尾が user でなければ ctx をトリム」しておくと API エラー耐性が上がる。
- **`messages.profiles` の埋め込み select の戻り型**: `profiles(display_name)` は Supabase の関係次第で配列にも単一にもなる。`as unknown as MessageRow[]` でキャストしているが、複数 profiles が紐づくケース (1:N) で実行時破綻するリスクは残る。型生成 (`supabase gen types`) を通すのが本筋。
