## レビュー結果

### バグ/エッジケース漏れ
- **`SUPABASE_SERVICE_ROLE_KEY` が未定義/空文字のときの挙動が危険**：環境変数取得部でフォールバックしていると、攻撃者が `Authorization: Bearer undefined` や `Bearer ` だけで通過する可能性がある。起動時（モジュールロード時）に値の存在をアサートすべき。
- **`OPTIONS` プリフライト**：Database Webhook では発生しないので実害なしだが、将来手動呼び出しを許す場合は preflight より先に 401 を返してしまう。今は問題なし。
- **`Authorization` ヘッダ末尾の空白・大文字小文字**：`Bearer` の `B` を小文字で送られたら不一致になる。Database Webhook は固定で `Bearer` を送るので実害はないが、ヘッダ名は HTTP 的には case-insensitive である点だけ留意。

### セキュリティ/入力検証
- **🔴 タイミング攻撃**：`authHeader !== \`Bearer ${KEY}\`` は短絡評価される文字列比較で、定数時間ではない。service_role key は長いので現実的リスクは小さいが、教科書的には NG。`crypto.subtle.timingSafeEqual` 相当（Deno なら `timingSafeEqual` from `std/crypto`）に置き換えるのが筋。
- **service_role key を Bearer で恒常的に流すこと自体の是非**：Supabase Database Webhook の HMAC 署名（`x-supabase-signature`）の方が漏洩耐性が高い。鍵がログ・プロキシ・転送ヘッダに残るリスクを M11 として懸念。次の段階で HMAC 化を検討推奨。
- **失敗ログなし**：401 を返した件数・送信元 IP を `console.warn` だけでも残しておくと、ブルートフォース検知の足がかりになる。
- **二重防御は正しい方針**：`verify_jwt` ON でも anon key 付き JWT は通過するため、service_role 限定にする本チェックは有意義。コメントの意図も明確。👍

### 設計/可読性
- **コメントが優秀**：migration 063/065 への参照、verify_jwt との関係明記で、半年後の自分が助かる構造。M3 評価。
- **抽出推奨**：認可ロジックは `assertServiceRoleAuth(req)` のような小関数に切り出した方が、本体の `try` 内が読みやすい。ただし今は1箇所のみなのでオーバーキルにもなる。任意。
- **`?? ""` は不要**：直後で `!==` 比較するので `null` のままでも結果は同じ。`?? ""` があると「空文字なら通るのか？」と一瞬読み手が止まる。削っても可。

### 優先度
- **今すぐ**：`SUPABASE_SERVICE_ROLE_KEY` の起動時バリデーション（未定義なら throw）。これは P0。
- **後で**：定数時間比較への置き換え、失敗時の最小限ログ。
- **任意**：HMAC 署名方式へ移行、認可関数の切り出し、`?? ""` 削除。

## Codex #2

同意点は、`SUPABASE_SERVICE_ROLE_KEY` の存在チェックが最重要です。もし `Deno.env.get(...) ?? ""` や同等のフォールバックなら、空キー時に `Authorization: Bearer ` で通る可能性があり、これはこの差分の最大リスクです。

見落としとしては、**401 が `try` の内側にある**点です。`SUPABASE_SERVICE_ROLE_KEY` が未定義で初期化時に throw する設計なら問題ありませんが、認証失敗をアプリ例外処理と混ぜると、将来 `catch` 側の変更で意図せず 500 化・ログ過多化する可能性があります。認証チェックは `try` の前でもよいです。

過剰指摘寄りなのはタイミング攻撃です。公開インターネット越しの Edge Function で service role key を文字列比較から現実的に復元するリスクはかなり低く、P0/P1 にするほどではないです。定数時間比較にする価値はありますが、「未設定時 fail closed」の方が圧倒的に優先です。

HMAC 化の提案は方向性として妥当ですが、この差分レビューではスコープ外寄りです。Supabase Database Webhook が service_role Bearer 前提で既に運用されているなら、今回の追加チェック自体は有効な防御強化です。

`?? ""` 削除は好みの範囲です。可読性なら `const expected = SUPABASE_SERVICE_ROLE_KEY;` を起動時に検証済みにして、ここでは単純比較する形が一番明確です。

## 同意点

- **「401 が `try` の内側」問題は的確**：見落としていた。catch 側の挙動変更で認証失敗が 500 に化けるリスクは現実的。`Deno.serve` のハンドラ冒頭、`try` の前に出すのが正解。これは即採用すべき。
- **タイミング攻撃の優先度下げ**：同意。`SUPABASE_SERVICE_ROLE_KEY` の起動時バリデーションが P0、定数時間比較は P3 で十分。順序を間違えないことの方が大事。
- **HMAC はスコープ外**：同意。この差分のレビュー範囲ではない。別チケットで扱う論点。

## 追加の論点

- **起動時バリデーションの形式**：モジュールトップで `if (!SUPABASE_SERVICE_ROLE_KEY) throw new Error(...)` にすると、Edge Function のコールドスタート時に即落ちる＝デプロイ直後に気付ける。ランタイム到達前に fail-fast させるのが Schneier 流（M11）。
- **`?? ""` は削るのに賛成**：起動時に存在保証されているなら、ハンドラ側は `req.headers.get("Authorization")` の `null` をそのまま `!==` 比較に流せばよい。読み手の認知負荷が下がる。

結論：`try` の外出し ＋ 起動時 throw、この2点が今回の差分への必須修正。残りは任意。
