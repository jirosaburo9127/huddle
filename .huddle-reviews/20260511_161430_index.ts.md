## レビュー

### バグ / エッジケース漏れ
- **DB 側の Webhook トリガー残置リスク (今すぐ確認)**: コードからは listen モードを削除したが、`messages` テーブルへの INSERT を起点とする Supabase Webhook / DB トリガーが残っていれば、Edge Function は呼ばれて `"ignored (listen mode removed)"` を返すだけになる。**無駄な invocation コストと latency が発生**するので、DB 側 (Dashboard の Webhooks か `supabase/migrations` 内のトリガー) も合わせて無効化するべき。差分の Function 側だけでは作業未完。
- **過去日ガードが LLM プロンプト任せ**: 「過去の日時なら呼ばない」をプロンプトのルール 3 で書いているが、LLM の遵守は 100% ではない。`propose_event` の tool_use を受け取った直後に `start_at_iso < now` ならコード側でも reject する二重防御を入れた方が堅い (元から無かった可能性あり、その場合は別タスクで OK)。
- **`MIKAN_USER_ID` 未設定時の挙動**: `mentioned_user_id !== MIKAN_USER_ID` の比較で MIKAN_USER_ID が undefined だと常に "not mikan" 返却になる。元から存在する問題なら今回の対象外。
- `LISTEN_COOLDOWN_MIN` を削った副作用として `event_proposals` テーブル参照も消えており、整合性は OK。

### 設計 / 可読性
- **`WebhookPayload` 型が緩くなった**: `MentionPayload | { type: string; table: string }` は「mentions 以外の何か」を表現するためだけのフォールバックで、型としては弱い。listen モードを完全に捨てる前提なら `type WebhookPayload = MentionPayload` にして、`table !== "mentions"` 分岐は `(payload as { table?: string }).table` で見るか、type guard 関数 (`isMentionPayload`) で narrowing するのが素直。`as MentionPayload` キャストも消える。
- **古いコメントの整理漏れ**: 上部ブロックコメントに「単一モード (mention)」と書きつつ、本文中の `// Mode A (mention) のみ受け付ける` のように「Mode A」表現が残っている。モード概念自体を廃止したのだから "Mode A" は消して「mentions テーブル INSERT のみ受け付ける」とした方が、半年後の読者が混乱しない。
- プロンプトの「呼ばずに確認質問を返すべきケース」は丁寧で良い。特に 4. (場所が論点になっているなら聞き返す / 一切話題に出ていないなら location 無しで OK) の使い分けは LLM が踏み外しがちな所なので明文化されているのは GOOD。

### セキュリティ / 入力検証
- **Webhook 署名検証なし (差分外だが指摘)**: この Edge Function は Supabase DB Webhook から呼ばれる想定だが、リクエスト元が本当に Supabase かを検証していない (diff 中に Authorization ヘッダ検証が見当たらない)。エンドポイントが推測可能だと、外部から偽 payload を送りつけて Claude API コール (= 課金) を発生させる攻撃が可能。Webhook secret か service_role JWT で gate するべき。M11 (シュナイアー) 案件。
- **`payload as MentionPayload` キャスト**: `table === "mentions"` だけでは型システム上 `record` の存在は保証されない。実運用上 Supabase は必ず record を送ってくるので落ちないはずだが、runtime 例外 → 500 でも捨てメッセージなので影響は小さい。前述の type guard に置き換えるとよい。
