## Claude #1

# コードレビュー: mikan-respond 認証ロジック差し戻し

## バグ / エッジケース漏れ
- **verify_jwt が本当に防御になるか未検証**: DB Webhook 側が空の `supabase.service_role_key` を渡しているなら、Authorization ヘッダ自体が空 or `Bearer ` の可能性が高い。だとすると `verify_jwt: true` でも 401 になるはず。「verify_jwt に委ねる」前提が成立しているか実機ログで確認すべき (もし通っていたら、それは verify_jwt が無効化されていることを意味する)。
- **payload の信頼性が消失**: 認証を抜いたため、後続の `payload.record` (チャネル ID・メッセージ内容など) を完全に信用したロジックになる。攻撃者が任意 INSERT イベントを偽装可能なら、AI 応答や DB 書き込みが攻撃面になる。
- **冪等性なし**: 同一 record_id の再送・リプレイで AI 呼び出しが多重発火しうる。Webhook はリトライが前提。

## 設計 / 可読性
- ロールバック理由をコメントに残しているのは◎。ただし「`current_setting(..., true)` が Cloud で空」という落とし穴は **feedback memory** に書く対象 (CLAUDE.md 3.5 のルール)。コメントは要約に留めて memory ID を貼るほうが資産になる。
- `TODO(security):` などのマーカーがあると、再導入忘れを grep で拾える。今のコメントは説明文で終わっており、未解決タスクとして検出しにくい。
- `SUPABASE_SERVICE_ROLE_KEY` 定数を読まなくなったなら、未使用 import / 定数を消すか、再導入時に残す意図をコメントで明示。

## セキュリティ / 入力検証
- **コスト/DoS リスク**: verify_jwt が anon key を通す設定なら、anon key (公開値) を知る誰でも LLM 呼び出しを誘発できる。レート制限・チャネル所在検証なしだと API コスト攻撃に晒される。
- **権限境界の格下げ**: 直前コミット (#5815) で導入したばかりの「service_role 厳格チェック」を、トリガー側を直す前に外したのは順序が逆。本来は **(1) DB トリガーで GUC を設定する migration → (2) Edge Function 側の厳格チェック維持** の順で進めるべき。今の状態は「壊れていたから鍵を外した」になっている。
- **WebhookPayload の型バリデーションなし**: `as WebhookPayload` キャストのみ。認証を外したなら、せめて record の shape / 期待 channel・workspace 所属の検証を加えるべき。

## 優先度
- **今すぐ**: 
  - verify_jwt が実際に Webhook 呼び出しを認可している証跡 (Supabase ログの 200/401) を確認。通っていなければそもそも復旧していない。
  - チャネル/ワークスペース所属チェック等、payload の最小バリデーションを追加 (認証ゼロ運用の応急処置)。
- **後で**: 
  - migration で `ALTER DATABASE ... SET supabase.service_role_key = ...` もしくは Vault から拾う形に直し、service_role Bearer チェックを再導入。
  - レート制限 (record_id ベース冪等キー + 短時間ウィンドウ) 。
- **任意**: 
  - 長文コメントを `TODO(security, ref: memory #5815):` + 1〜2 行に圧縮し、詳細は feedback memory へ。

同意点:

- **anon key が通る可能性は重要論点**です。Supabase Edge Function の `verify_jwt` は anon/service_role いずれの JWT も有効なら通す理解なので、anon key が公開前提である以上、これだけでは webhook 専用認可になりません。
- **コメントが実装上の未解決セキュリティ課題を埋もれさせている**点も同意です。最低でも `TODO(security): restore service_role-only webhook auth` のように grep 可能にすべきです。

見落とし気味の論点:
- `SUPABASE_SERVICE_ROLE_KEY` がこの関数内で他用途に使われていないなら、今回の差分で **未使用 env 読み込み・未使用定数化していないか**は実際に確認したいです。Deno lint で落ちる可能性があります。
- 「GUC を設定する」は注意が必要です。`ALTER DATABASE ... SET supabase.service_role_key = ...` は秘密値を DB 設定に置く設計なので、運用・漏えい面で微妙です。Supabase Vault / pg_net のヘッダ設定 / webhook 設定側で確実に secret を渡す方向の方が筋が良い可能性があります。
- 厳格チェックを戻すなら、単純な `Authorization === Bearer ${SERVICE_ROLE_KEY}` だけでなく、**共有 secret ヘッダ**の方が webhook 用途として分離しやすいです。service_role JWT を webhook 認証トークンとして兼用する設計自体も再考余地があります。

過剰かもしれない指摘:
- **冪等性なし**は正しいが、この差分固有の退行ではない可能性が高いです。今回の認証差し戻しレビューでは P1 ではなく、別課題扱いでよさそうです。
- **WebhookPayload の型バリデーションなし**も以前からなら、この変更の直接責任としてはやや広いです。ただし認証を弱めるなら優先度は上がります。
- 「verify_jwt が通っていたら無効化されている意味」は少し言い切り過ぎです。anon/service_role JWT が渡っていれば `verify_jwt: true` でも通ります。問題は「通ること」ではなく「誰の JWT なら通るのか」です。
