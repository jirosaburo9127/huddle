## Claude #1

# レビュー: mikan-respond/index.ts (webhook secret 強化)

## バグ / エッジケース
- `Deno.env.get()` の戻り値は `string | undefined`。`if (!MIKAN_WEBHOOK_SECRET) throw` のあとは TS が `string` に narrowing するはずだが、`const` でモジュールトップレベルなのでクロージャ越しでも narrowing が維持されるか要確認 (Deno/TS 5.x なら OK)。心配ならアサーション後に `const SECRET: string = MIKAN_WEBHOOK_SECRET` と再代入が無難。
- 環境変数に空白文字だけが入っているケース (`"   "`) は通ってしまう。`.trim().length > 0` でガードしておくと運用ミスを早期に拾える。
- module init で throw する設計は Edge Function のコールドスタートで全リクエストを 500 にする。秘密ローテーション中の一時的な未設定でも全停止する点は「意図通り」と明記しておくと良い (コメントに「fail-fast 採用」とあるので OK)。

## 設計 / 可読性
- `MIKAN_SECRET_HEADER` を定数化したのは良いが、**diff には宣言しか見えない**。検証側 (request handler) で実際に `req.headers.get(MIKAN_SECRET_HEADER)` を使っているか要確認。使っていなければ typo 防止の目的を果たさない。
- ヘッダ名は HTTP 仕様上 case-insensitive なので `Headers.get()` ならどの大小でも引ける。ただし他システムとの相互運用や文書化のために大文字小文字を固定するのは妥当。
- コメントが少し冗長。`?? ""` のリスク説明はコミットメッセージに移し、コードコメントは `// MIKAN_WEBHOOK_SECRET は起動時必須` 程度で十分。

## セキュリティ / 入力検証
- **timing-safe 比較**: diff には現れていないが、検証側で `headerValue === MIKAN_WEBHOOK_SECRET` を使っているなら timing attack の余地あり。webhook の現実的なリスクは低いが、`crypto.subtle` の constant-time 比較 (もしくは長さチェック + ループ) に置き換える価値あり。
- **空ヘッダの扱い**: `req.headers.get(MIKAN_SECRET_HEADER)` は未設定時 `null` を返す。`null === ""` は false なので問題なし。ただし環境変数の `trim` 抜けと組み合わせると、空白だけのヘッダで一致する事故が起こり得る。
- secret の長さ要件 (最小 32 bytes など) を init 時にチェックしておくと、弱い値が混入したときに気付ける。

## 優先度
- **今すぐ**: 検証側で `MIKAN_SECRET_HEADER` 定数を実際に使っているか確認 (使われていなければ宣言だけ残しても意味がない)。
- **後で**: 比較を timing-safe に置き換え、secret の長さ・空白チェックを init に追加。
- **任意**: コメントの圧縮、`SECRET: string` への再束縛で TS narrowing を明示。

全体としては `?? ""` フォールバック撤廃は明確な改善。あとは「定数を使う側」と「比較方法」の二点で詰めれば完成度が上がります。

- 既存テストがあるなら「env 未設定時に起動失敗」「ヘッダ未設定時 reject」「正しい secret で accept」の回帰テストが一番実益あります。

- timing-safe 比較は理屈としては正しいですが、DB trigger から Edge Function への共有 secret 検証なら優先度は低め。外部公開 webhook として攻撃試行される前提なら検討、くらいでよいです。
