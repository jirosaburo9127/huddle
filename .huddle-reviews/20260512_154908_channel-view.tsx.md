## Codex レビュー

- [優先度: 今すぐ] `insert` の `user_id` に `currentUserId` を使う前に、`getUser()` の `user.id` と一致することを検証してください。`currentUserId` が stale / 誤注入された場合に、送信失敗や RLS 設定次第でなりすましリスクになります。`if (user.id !== currentUserId) { rollback; return; }` を入れるか、DB 挿入は `user.id` を使ってください。

- [優先度: 今すぐ] `supabase.auth.getUser()` が失敗した場合だけでなく、後続の `messages.insert(...)` が失敗した場合にも `sentMessageIdsRef.current.delete(newMessageId)` と optimistic message の rollback が必要です。これが無いと、送信失敗したメッセージが画面に残り、同 ID の realtime も除外され続けます。

- [優先度: 後で] realtime の後追い hydrate は await 後に無条件で `setMessages` しています。チャンネル遷移や unmount 後に古い callback が戻る可能性があるため、`payload.new.channel_id === channel.id` の再確認、または `isMounted/currentChannelId` ref でガードすると安全です。

- [優先度: 任意] optimistic profile の `email` / `display_name` が空文字固定になったため、`myProfile` 未取得時に送信者名が一時的に空表示になる可能性があります。UI 側に fallback が無いなら、少なくとも `display_name: myProfile?.display_name || "You"` のような表示崩れしない値にしてください。
